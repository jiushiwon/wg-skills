# 多原型图梳理指引

当用户一次性发来多张原型图时，不要直接逐张复刻，先梳理关系。

## 为什么需要先梳理

- 多张图之间可能有公共组件（导航栏、tab bar、按钮样式）
- 页面之间有跳转关系，需要统一路由
- 同一个组件在不同页面可能有细微差异，需要决定是统一还是变体
- 复刻顺序会影响效率：先公共组件，再具体页面

## 梳理步骤

### 1. 给每张图命名

根据图中标题或功能，给每张图一个页面名：

```
图 1: 首页 (home)
图 2: 会议详情页 (meeting-detail)
图 3: 我的会议列表 (my-meetings)
图 4: 个人中心 (profile)
```

### 2. 识别页面类型

| 页面 | 类型 | 路由建议 |
|---|---|---|
| 首页 | Tab 页 | /pages/index/index |
| 会议详情 | 二级页 | /pages/meeting/detail |
| 会议列表 | Tab 页或二级页 | /pages/meeting/list |
| 个人中心 | Tab 页 | /pages/profile/index |

### 3. 提取公共组件

跨页面出现时，判定为公共组件：

- 顶部导航栏
- 底部 tab bar
- 按钮样式
- 卡片样式
- 列表项样式
- 图标风格

**原则**：3 个及以上页面出现的元素，必须沉淀为公共组件。

### 4. 梳理跳转关系

用箭头表示页面间跳转：

```
首页 ──点击会议卡片──→ 会议详情页
会议详情页 ──点击返回──→ 首页
底部 tab: 首页 ⇄ 我的会议 ⇄ 个人中心
```

### 5. 输出复刻计划

```
第一步：沉淀公共组件
  - CustomNavBar
  - TabBar
  - ActionButton
  - MeetingCard

第二步：逐个复刻页面
  - 首页
  - 会议详情页
  - 我的会议列表
  - 个人中心

第三步：串联路由和状态
  - 配置 pages.json / 路由表
  - 接入会议数据 mock
```

## 与用户确认

梳理完成后，先给用户看计划，确认以下问题：

1. 页面命名和路由是否符合预期？
2. 哪些元素需要作为公共组件？
3. 是否有图中未体现的跳转关系？
4. 是否需要优先复刻某个页面？

**不要直接开始写代码**，先让用户确认梳理结果。

## 多图时的交付形态

| 场景 | 输出 |
|---|---|
| 用户只想先看效果 | 每张图输出一个单文件 HTML |
| 用户要接入 uniapp/Vue 项目 | 先输出公共组件，再输出页面 |
| 用户要给开发团队 | 输出组件清单 + 页面结构 + 路由表 + 代码 |

## 注意事项

- 不同图的色调可能不一致，需要收敛到同一套 Token
- 同一个按钮在不同页面尺寸可能不同，优先统一
- 页面状态（空态/加载态/错误态）按业务补充
- 先完成所有页面的"合格"标准，再挑重点页面优化到"优秀"
