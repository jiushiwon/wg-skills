# 风格校准（Voice Calibration）

本文档定义风格校准功能，让 AI 模仿用户的个人写作风格。

---

## 使用方式

用户提供自己的写作样本，AI 分析其风格特征后，用该风格重写文本。

### 内联提供样本

```
检测这段文案的 AI 风，并用我的风格重写。
这是我的写作样本：
[样本文本]
```

### 从文件提供样本

```
检测这段文案的 AI 风，参考我的写作风格。
我的写作样本在：path/to/my-writing.md
```

---

## 风格分析维度

### 1. 句子长度模式

| 类型 | 特征 | 示例 |
|------|------|------|
| 短促型 | 5-10 词，节奏快 | "This works. Period." |
| 流畅型 | 20-30 词，绵长 | "When we consider the implications of this approach, we find that..." |
| 混合型 | 长短交替 | 短句 + 长句 + 短句 |

### 2. 词汇水平

| 类型 | 特征 |
|------|------|
| 口语化 | "stuff", "things", "kinda", "sorta" |
| 学术型 | "demonstrate", "indicate", "significant" |
| 中间型 | 日常词为主，偶尔专业术语 |

### 3. 段落开头方式

| 类型 | 特征 |
|------|------|
| 直入型 | 直接抛出观点或事实 |
| 铺垫型 | 先给上下文再展开 |
| 反问型 | 以问题开头 |

### 4. 标点习惯

| 特征 | 说明 |
|------|------|
| 破折号爱好者 | 频繁使用 — 做插入语 |
| 括号狂魔 | 频繁用 () 补充说明 |
| 分号使用者 | 用 ; 连接相关句子 |
| 句号极简 | 短句为主，很少逗号 |

### 5. 口头禅/重复语

| 类型 | 示例 |
|------|------|
| 填充词 | "you know", "honestly", "frankly" |
| 转折词 | "but", "however", "though" 的偏好 |
| 强调词 | "really", "actually", "literally" |

### 6. 过渡方式

| 类型 | 特征 |
|------|------|
| 显式连接 | "Furthermore", "In addition", "On the other hand" |
| 隐式跳跃 | 直接开始下一段，无明显过渡 |
| 场景转换 | 用时间/地点转换 |

---

## 风格模仿规则

### 必须匹配的维度

1. **句子长度** - 如果用户写短句，不要产长句
2. **词汇等级** - 如果用户用 "stuff"，不要升级为 "elements"
3. **段落结构** - 匹配用户的段落长度和开头方式
4. **标点偏好** - 如果用户不用破折号，不要加
5. **过渡风格** - 匹配用户的过渡习惯

### 可选增强的维度

1. **个人观点** - 如果用户文本有观点，新文本也应有
2. **幽默感** - 如果用户文风幽默，新文本也应有趣
3. **情感表达** - 匹配用户的情感直接程度

---

## 风格校准流程

```
用户提供文本 + 可选写作样本
    ↓
检测 AI 风（原有功能）
    ↓
如果有样本：
  分析样本风格特征
  用该风格重写
如果没有样本：
  用默认自然风格重写
    ↓
输出检测报告 + 改写后的文本
```

## 示例

### 用户写作样本

> 我其实不太确定这件事。你知道吗，有时候就是这样，你以为搞懂了，结果发现根本没搞懂。不过没关系，继续试呗。

### 分析结果

| 维度 | 特征 |
|------|------|
| 句子长度 | 短到中等，混合 |
| 词汇 | 口语化，"呗", "嘛" 等语气词 |
| 段落开头 | 直入，个人观点 |
| 过渡 | "不过", "其实" 等口语化转折 |
| 情感 | 直接，略带自嘲 |

### AI 原文

> 综上所述，该方法具有显著的优越性，能够有效提升系统性能。

### 用用户风格重写

> 说实话，这个方法确实挺管用的。你知道吗，我一开始也没觉得能行，结果试了一下发现还真不错。不过具体能提升多少，还得再看。

---

## 注意事项

1. **样本长度** - 建议提供 200-500 字的样本，太短无法准确分析
2. **样本质量** - 样本应是用户自己写的，不能是 AI 生成的
3. **内容类型** - 技术文档和散文的风格不同，尽量提供同类型样本
4. **混合风格** - 如果用户有多种写作风格，可以分别提供样本
